LESSON 01 — INTRODUCTION
WHAT IS CI/CD?
CI/CD is the practice of automating the steps between writing code and running it in production. Instead of manually building, testing, and deploying — every code push triggers an automated pipeline that does it for you.
THE PAIN WITHOUT CI/CD
  Developer workflow WITHOUT CI/CD:
  1. Write code locally
  2. "Works on my machine!" ✅
  3. Manually build JAR: ./gradlew bootJar (hope it works)
  4. SCP file to server: scp human.jar user@server:/opt/human/
  5. SSH to server: ssh user@server
  6. Stop old app: pkill -f "human.jar"
  7. Start new app: java -jar /opt/human/human.jar &
  8. Manually test: curl http://server/api/health
  9. 3 hours later: "Why is prod down??" 💥
  10. Hotfix... repeat from step 1

  Developer workflow WITH CI/CD:
  1. Write code
  2. git push origin main
  3. ✅ Automated: build → test → docker build → deploy → health check
  4. Get Slack notification: "✅ Deployed human v1.4.2 to production"
  5. Go drink coffee ☕
⚡
Speed
Deploy multiple times per day instead of once per sprint. Smaller releases = smaller risk.
🛡️
Confidence
Automated tests catch bugs before they reach production. Every push is validated.
🔁
Consistency
Every deployment follows the exact same steps. No more "I forgot to run migrations."
👥
Collaboration
Team members can deploy safely. No single "deployment wizard" bottleneck.
🔍
Visibility
Every deploy is logged, traceable, and auditable. Know exactly what changed and when.
↩️
Rollback
Automated rollback on failure. Production recovers in seconds, not hours.

A) They need better developers

LESSON 02 — CONCEPTS
CI vs CD vs CD
There are TWO different "CD"s. Most people confuse them. Understanding the difference between Continuous Integration, Continuous Delivery, and Continuous Deployment is fundamental.
THE FULL CI/CD PIPELINE
  ── CONTINUOUS INTEGRATION (CI) ──────────────────────────────
  Developer pushes code
       │
       ▼
  ┌─────────────────────────────────────────────────────────┐
  │  Automated: Build → Unit Tests → Integration Tests      │
  │  Code Quality → Security Scan → Coverage Report         │
  └─────────────────────────────────────────────────────────┘
  Goal: Detect bugs FAST. Merge small changes often.

  ── CONTINUOUS DELIVERY (CD #1) ──────────────────────────────
       │  (after CI passes)
       ▼
  ┌─────────────────────────────────────────────────────────┐
  │  Automated: Docker Build → Push to Registry             │
  │  Deploy to Staging → Run E2E Tests → Ready for prod     │
  └──────────────────────┬──────────────────────────────────┘
                         │ 👆 MANUAL APPROVAL REQUIRED
                         ▼
                    Production Deploy

  Goal: Code is ALWAYS deployable. Human decides WHEN to release.

  ── CONTINUOUS DEPLOYMENT (CD #2) ────────────────────────────
       │  (after all tests pass — NO human approval)
       ▼
  ┌─────────────────────────────────────────────────────────┐
  │  Automated: Everything above + Auto-deploy to prod      │
  │  Health check → Rollback if unhealthy                   │
  └─────────────────────────────────────────────────────────┘
  Goal: Every passing commit automatically reaches production.
  Used by: Netflix, Facebook, Amazon (thousands of deploys/day)
PracticeAutomation LevelWho triggers prod deploy?Best for
Continuous IntegrationBuild + Test autoManualAll teams — minimum viable CI/CD
Continuous DeliveryBuild + Test + Stage autoHuman approvesEnterprise, regulated industries
Continuous DeploymentEverything automatedNo one — fully automatedHigh-trust, well-tested products

LESSON 03 — GIT STRATEGY
GIT BRANCHING
STRATEGY
Your branching strategy determines how CI/CD pipelines trigger and what gets deployed where. The right strategy prevents chaos on a team of any size.
GITFLOW — MOST POPULAR FOR ENTERPRISE
  main     ────●──────────────────────────●────── (production)
                  ↑                          ↑
  release      ●──●──●                  ●──●──●   (release candidate)
                ↑                        ↑
  develop  ──●──●──●──●──●──●──●──●──●──●──     (integration)
               ↑     ↑     ↑
  feature  ●──●  ●──●  ●──●                    (your feature work)

  hotfix   ────────────────●──●                 (emergency fixes)

  Pipeline triggers:
  feature/* push   → CI only (build + test)
  develop push     → CI + deploy to DEV environment
  release/* push   → CI + deploy to STAGING environment
  main push        → CI + deploy to PRODUCTION ✅
TRUNK-BASED DEVELOPMENT — USED BY GOOGLE, FACEBOOK
  main  ──●──●──●──●──●──●──●──●──●──●──  (everyone pushes here!)
             ↑  ↑  ↑  ↑
  Short-lived feature branches (max 1-2 days)
  Merge to main multiple times per day
  Feature flags control visibility in production

  Pros: Simple, fast feedback, no merge hell
  Cons: Requires excellent test coverage and feature flags
ℹ️ For your Demo App Spring Boot project, GitFlow is recommended: main → production, develop → staging, feature/* → dev testing. Simple and battle-tested for enterprise Java apps.

LESSON 08 — SECRETS
SECRETS &
ENVIRONMENTS
Never put passwords, API keys, or tokens in your workflow files. GitHub Secrets encrypts them and injects them securely at runtime.
3 LEVELS OF SECRETS IN GITHUB
  Repository Secrets  (Settings → Secrets → Actions)
  Available to all workflows in the repo.
  Use for: DOCKER_PASSWORD, SSH_PRIVATE_KEY, SONAR_TOKEN

  Environment Secrets  (Settings → Environments → staging/production)
  Only available to workflows targeting that environment.
  Can require manual approval before use!
  Use for: PROD_DB_PASSWORD, PROD_SERVER_IP (different per env)

  Organization Secrets  (Organization → Settings → Secrets)
  Shared across multiple repositories.
  Use for: shared Docker Hub credentials, SonarQube token
.github/workflows/deploy.yml — using secrets + environments
🚨 NEVER do run: echo "${{ secrets.MY_SECRET }}" — this prints the secret in the log (even though GitHub masks it, it's a bad habit). Use secrets only in commands that need them.

LESSON 09 — DEPLOYMENT
DEPLOY TO
SERVER (SSH)
The most common deployment pattern for teams without Kubernetes — SSH into your production server, pull the new Docker image, and restart the container.
SSH DEPLOYMENT FLOW
  GitHub Actions Runner
       │
       │  SSH with private key
       ▼
  Production Server (your VPS/cloud instance)
       │
       ├── docker pull ghcr.io/org/human:sha-abc1234
       ├── docker stop demo-app
       ├── docker rm demo-app
       ├── docker run -d --name demo-app ...new image...
       └── curl http://localhost:9191/actuator/health → ✅ UP

  Setup:
  1. Generate SSH key pair: ssh-keygen -t ed25519 -f deploy_key
  2. Add PUBLIC key to server: ~/.ssh/authorized_keys
  3. Add PRIVATE key to GitHub: Settings → Secrets → SSH_PRIVATE_KEY

LESSON 13 — DEPLOY STRATEGIES
DEPLOYMENT
STRATEGIES
HOW you deploy matters as much as WHAT you deploy. Different strategies offer different tradeoffs between risk, downtime, and rollback speed.
DEPLOYMENT STRATEGY COMPARISON
  1. Big Bang / Recreate
  Stop v1 → Deploy v2 → Start v2
  Downtime: YES ⚠️  Rollback: Redeploy v1 (slow)
  Use: Dev environments only

  2. Rolling Update
  Instance: [v2] [v1] [v1] → [v2] [v2] [v1] → [v2] [v2] [v2]
  Downtime: NO  Rollback: Redeploy v1 (medium speed)
  Use: Standard production updates

  3. Blue-Green
  Blue (v1) = LIVE          Green (v2) = IDLE
  Deploy v2 to Green → Test → Switch load balancer to Green
  Blue becomes standby for instant rollback!
  Downtime: NO  Rollback: Switch back to Blue (INSTANT!) ✅
  Use: Critical services, zero-risk deploys

  4. Canary Release
  5% traffic → v2  (watch metrics for 30 min)
  10% traffic → v2  (still healthy?)
  50% → v2  (looks good!)
  100% → v2  (full rollout)
  Rollback: Route 100% back to v1
  Use: High-traffic apps, A/B testing features

LESSON 16 — SECURITY
SECURITY IN
CI/CD
Your CI/CD pipeline is a high-privilege system with access to production servers, databases, and registries. Securing it is non-negotiable.
1
Pin action versions with SHA hashes
Use actions/checkout@abc123def456 not @v4. An attacker could push malicious code to a tag. SHA is immutable.
2
Use OIDC instead of long-lived secrets for cloud
GitHub Actions can authenticate to AWS/GCP/Azure using short-lived OIDC tokens instead of static API keys stored as secrets.
3
Scan Docker images for vulnerabilities
Add Trivy or Snyk scan to your pipeline. Block deployment if CRITICAL vulnerabilities found in your image.
4
Least privilege permissions
Set permissions: read-all at workflow level, then explicitly grant only what each job needs.
5
Audit your secrets regularly
Rotate credentials every 90 days. Use GitHub's secret scanning to detect if secrets are accidentally committed.
security scanning in pipeline
🚨 The most common CI/CD security mistake: storing database passwords or SSH keys as plaintext in workflow files or scripts checked into the repository. Always use GitHub Secrets — never hardcode credentials.
