LESSON 01 — INTRODUCTION
What is Docker?
Docker packages your application and ALL its dependencies into a portable unit called a container. It runs identically on any machine — your laptop, a test server, or a production cloud server.
😤
The Problem
"Works on my machine!" — app works locally but breaks on server due to different Java versions, OS, or configs.
📦
Docker Solution
Bundle app + JDK + OS libs + config into one container. Runs identically everywhere.
🚀
vs Virtual Machine
VM = full OS (GBs, minutes to start). Container = shares host OS kernel (MBs, milliseconds to start).
🔁
Reproducible
Same image always produces same result. No more "it worked last week" mysteries.
DOCKER vs VIRTUAL MACHINE
  Virtual Machine                    Docker Container

  ┌──────────────────────┐           ┌──────────────────────┐
  │   App A  │   App B   │           │   App A  │   App B   │
  ├──────────┼───────────┤           ├──────────┼───────────┤
  │  Guest   │  Guest    │           │ Container│ Container │
  │   OS     │   OS      │           │  (libs)  │  (libs)   │
  ├──────────┴───────────┤           ├──────────┴───────────┤
  │      Hypervisor      │           │     Docker Engine     │
  ├──────────────────────┤           ├──────────────────────┤
  │      Host OS         │           │      Host OS          │
  ├──────────────────────┤           ├──────────────────────┤
  │      Hardware        │           │      Hardware         │
  └──────────────────────┘           └──────────────────────┘
  Size: ~10GB  Start: ~60s           Size: ~200MB  Start: <1s
ℹ️ Docker uses Linux namespaces and cgroups to isolate containers. Each container gets its own filesystem, network, and process space — but shares the host OS kernel.

LESSON 03 — CONCEPTS
Images vs Containers
This is the most important concept in Docker. Understanding the difference between an image and a container is fundamental.
IMAGE vs CONTAINER ANALOGY
  Image  =  A blueprint / recipe / class definition
  Container  =  A running instance created from that blueprint

  Real world analogy:
  ┌─────────────────────────────────────────────────────┐
  │  Image = Cookie Cutter 🍪                           │
  │  Container = The actual cookie made from it         │
  │                                                     │
  │  One image → can create MANY containers             │
  │  nginx:latest → container1, container2, container3  │
  └─────────────────────────────────────────────────────┘

  Image Layers (read-only):
  ┌─────────────────────┐
  │  Your App JAR       │  ← Layer 4 (your changes)
  ├─────────────────────┤
  │  application.yml    │  ← Layer 3
  ├─────────────────────┤
  │  JDK 21             │  ← Layer 2
  ├─────────────────────┤
  │  Ubuntu base        │  ← Layer 1 (FROM ubuntu)
  └─────────────────────┘
  + Writable container layer on top when running
📸
Image
Read-only template. Stored in registry (Docker Hub). Built from Dockerfile. Can be shared and versioned.
🏃
Container
Running instance of an image. Has writable layer. Isolated process. Can be started, stopped, deleted.
🏪
Registry
Storage for images. Docker Hub is public. You can run private registries. Like GitHub but for Docker images.
🏷️
Tag
Version label for an image. nginx:latest, nginx:1.25, java:21-slim. "latest" is default if not specified.

LESSON 05 — DOCKERFILE
Dockerfile Basics
A Dockerfile is a text file with instructions to build a Docker image. Each instruction creates a new layer in the image.
DOCKERFILE INSTRUCTIONS REFERENCE
  FROM        Base image to start from           FROM eclipse-temurin:21-jdk-alpine
  WORKDIR     Set working directory              WORKDIR /app
  COPY        Copy files from host to image      COPY target/app.jar app.jar
  ADD         Like COPY but can unzip/download   ADD app.tar.gz /app
  RUN         Execute command during build       RUN apt-get install -y curl
  ENV         Set environment variable           ENV JAVA_OPTS="-Xmx512m"
  ARG         Build-time argument                ARG VERSION=1.0
  EXPOSE      Document port (does NOT publish)   EXPOSE 9191
  VOLUME      Create mount point for data        VOLUME /var/log
  CMD         Default command when container runs CMD ["java","-jar","app.jar"]
  ENTRYPOINT  Main command (CMD appends to it)   ENTRYPOINT ["java"]
  HEALTHCHECK Check if container is healthy      HEALTHCHECK CMD curl -f http://localhost/health
  USER        Switch to non-root user            USER appuser
  LABEL       Add metadata                       LABEL version="1.0"

LESSON 07 — ADVANCED DOCKERFILE
Multi-stage Builds
Multi-stage builds let you use multiple FROM statements. Build in one stage, copy only the output to the final tiny image. This is the standard for production Spring Boot Demo Apps.
MULTI-STAGE BUILD CONCEPT
  Stage 1: builder                    Stage 2: final
  ┌─────────────────────┐              ┌─────────────────────┐
  │  FROM gradle:8-jdk21│              │ FROM eclipse-temurin│
  │  COPY source code   │              │       :21-jre-alpine│
  │  RUN gradle bootJar │──── JAR ────►│ COPY --from=builder │
  │  (includes gradle,  │              │ CMD java -jar       │
  │   source, tests...) │              │                     │
  │  Size: ~800MB       │              │  Size: ~180MB  ✅   │
  └─────────────────────┘              └─────────────────────┘

LESSON 08 — NETWORKING
Docker Networking
Containers are isolated. Docker networking lets containers talk to each other securely, and controls what's exposed to the outside world.
DOCKER NETWORK TYPES
  bridge (default)     Containers on same network can talk by name
  host                Container shares host's network directly
  none                No network access (complete isolation)
  overlay             Multi-host networking (Docker Swarm)

LESSON 09 — DOCKER COMPOSE
Docker Compose Intro
Docker Compose lets you define and run multi-container applications using a single YAML file. Instead of running 5 docker run commands, you run one: docker compose up.
WITHOUT vs WITH DOCKER COMPOSE
  Without Compose — 5 commands to remember:
  docker network create human-net
  docker run -d --name oracle --network human-net -e PASS=... oracle/database:19.3.0-ee
  docker run -d --name redis --network human-net redis:7-alpine
  docker run -d --name jaeger --network human-net -p 16686:16686 jaegertracing/all-in-one
  docker run -d --name human --network human-net -p 9191:9191 -e DB_URL=... human:latest

  With Compose — just one command:
  docker compose up -d   ✅  (everything defined in docker-compose.yml)
📋
One File
All services, networks, volumes defined in one docker-compose.yml. Version controlled with your code.
🔗
Auto Networking
Compose creates a network automatically. Services talk to each other by service name.
📊
Dependency Order
Use depends_on to start services in the right order (DB before app).
♻️
Easy Lifecycle
up, down, restart, logs, ps — manage entire stack with simple commands.
CommandDescription
docker compose up -dStart all services in background

LESSON 11 — DATA PERSISTENCE
Volumes & Data
Containers are ephemeral — when you remove a container, its data is gone. Volumes persist data outside the container lifecycle.
3 WAYS TO PERSIST DATA
  1. Named Volumes  (Recommended for databases)
     docker run -v oracle-data:/opt/oracle/oradata oracle-db
     Data stored in Docker-managed location. Survives container deletion.

  2. Bind Mounts   (Recommended for config & logs)
     docker run -v /host/path:/container/path myapp
     Maps a host directory into container. You control location.
     Use for: logs, config files, development code hot-reload

  3. tmpfs Mounts  (Sensitive data in memory only)
     docker run --tmpfs /run:rw,size=64m myapp
     Stored in host RAM only. Never written to disk.
     Use for: secrets, session data

  ┌──────────────────────────────────────────────────────┐
  │               Host Machine                           │
  │  /var/lib/docker/volumes/oracle-data/  ◄──── Volume │
  │  /home/user/human/logs/               ◄──── Bind    │
  └──────────────────────────────────────────────────────┘

LESSON 13 — PRODUCTION SECURITY
Security Best Practices
Docker containers are not automatically secure. Follow these practices to harden your production setup.
1
Never run as root
Add a non-root user in your Dockerfile. Running as root means a container exploit could compromise the host.
2
Use specific image tags, never :latest in production
Use eclipse-temurin:21.0.4-jre-alpine not :latest. Latest can break without warning.
3
Use .env files for secrets, not hardcoded values
Keep passwords in .env files. Add .env to .gitignore. Never commit passwords to git.
4
Don't expose unnecessary ports
Oracle DB on 1521 should never be public. Only expose what clients actually need to reach.
5
Use read-only mounts where possible
Mount config files with :ro flag so containers can't modify them.
