LESSON 01 — INTRODUCTION
WHAT IS
KUBERNETES?
Kubernetes (K8s) is an open-source container orchestration system. It automates deploying, scaling, and managing containerized applications across a cluster of machines. Born at Google in 2014, it now runs the majority of the world's cloud infrastructure.
THE PROBLEM KUBERNETES SOLVES
  Without Kubernetes (Docker alone):
  Server 1: docker run demo-app:v1  (running)
  Server 2: docker run demo-app:v1  (running)
  Server 3: docker run demo-app:v1  (crashed at 2am — nobody noticed 😴)

  Questions you can't answer:
  • Which server has the newest version?
  • How do I update all 3 without downtime?
  • What happens when traffic spikes to need 10 servers?
  • How do I roll back if the update breaks prod?

  With Kubernetes:
  You declare: "I want 3 replicas of demo-app:v2, always running, 512MB RAM each"
  K8s handles: scheduling, restarts, rolling updates, scaling, self-healing
  You sleep peacefully. ☕
🔄
Self-Healing
Container crashes? K8s restarts it automatically. Node dies? Pods migrate to healthy nodes.
📈
Auto-Scaling
CPU high? K8s adds more pods automatically. Traffic drops? Scales back down.
🔀
Rolling Updates
Deploy new version with zero downtime. Automatic rollback if health checks fail.
⚖️
Load Balancing
Distributes traffic across all healthy pods. Built-in service discovery.
💾
Storage Management
Automatically provisions and mounts storage for stateful apps like databases.
🌍
Multi-cloud
Same YAML files work on AWS, GCP, Azure, on-premise. No vendor lock-in.

Docker vs Kubernetes — When do you need K8s?

LESSON 04 — CORE OBJECTS
PODS
A Pod is the smallest deployable unit in Kubernetes. It wraps one or more containers that share the same network and storage. Think of it as a "wrapper" around your Docker container.
POD ANATOMY
  ┌─────────────────── Pod ──────────────────────────────┐
  │                                                      │
  │  ┌───────────────────────┐  ┌─────────────────────┐  │
  │  │  Main Container       │  │  Sidecar Container  │  │
  │  │  (demo-app:v1.4.2)       │  │  (otel-collector)   │  │
  │  │  port: 9191           │  │  port: 4317         │  │
  │  └───────────────────────┘  └─────────────────────┘  │
  │                                                      │
  │  Shared network: localhost (containers talk on 127.x)│
  │  Shared volumes: /app/logs, /tmp                     │
  │  Shared IP: 10.244.1.5 (cluster-internal)            │
  │  Lifecycle: all containers start/stop together       │
  └──────────────────────────────────────────────────────┘
  Pods are EPHEMERAL — never assume a Pod's IP is stable!

Pod Lifecycle

PhaseMeaning
PendingPod accepted by K8s but not yet scheduled or container images downloading
RunningPod bound to a node, all containers started
SucceededAll containers exited with code 0 (batch jobs)
FailedAt least one container exited non-zero
CrashLoopBackOffContainer keeps crashing — K8s waits before restarting (exponential backoff)
ImagePullBackOffCannot pull the container image (wrong name, auth needed)
ℹ️ Never run bare Pods in production. Use Deployments instead. If a Pod dies (node failure, crash), K8s does NOT restart it unless it's managed by a controller like a Deployment.

Readiness vs Liveness Probes

ProbePurposeFailure action
readinessProbeIs the app ready to serve traffic? (Spring Boot still warming up?)Remove Pod from Service load balancer — no traffic sent
livenessProbeIs the app still alive? (deadlocked? OOM?)Kill and restart the container
startupProbeHas the app finished starting? (for slow starters)Kill and restart — disables liveness during startup
💡 Spring Boot Actuator provides the perfect probe endpoints out of the box. Add management.health.livenessState.enabled=true and management.health.readinessState.enabled=true to your application.properties.

LESSON 06 — CORE OBJECTS
SERVICES
Pods die and get new IPs constantly. A Service gives you a stable DNS name and IP that load-balances to matching Pods — regardless of how many come and go.
4 SERVICE TYPES
  ClusterIP (default)
  Stable internal IP. Only accessible WITHIN the cluster.
  Use for: app → database, app → app communication

  NodePort
  Opens a port (30000-32767) on EVERY node.
  Access: http://node-ip:30080 from outside
  Use for: development, testing

  LoadBalancer
  Provisions a cloud load balancer (AWS ELB, GCP LB).
  Gets external IP automatically.
  Use for: production internet-facing services

  ExternalName
  Maps service to external DNS (e.g. your Oracle DB on another server)
  Use for: pointing to resources outside K8s

LESSON 09 — CONFIGURATION
CONFIGMAPS
& SECRETS
ConfigMaps store non-sensitive config. Secrets store sensitive data (passwords, tokens) encoded in base64. Both are injected into pods as env vars or mounted as files.
configmap.yaml + secret.yaml
🚨 Kubernetes Secrets are only base64 encoded, not encrypted! Anyone with cluster access can decode them. For production, use External Secrets Operator with HashiCorp Vault, AWS Secrets Manager, or seal secrets with SealedSecrets.
LESSON 10 — STORAGE
VOLUMES &
PERSISTENT STORAGE
Containers are ephemeral — data dies with them. Kubernetes Volumes, PersistentVolumes, and PersistentVolumeClaims give your apps durable storage.
STORAGE HIERARCHY
  StorageClass  →  defines HOW storage is provisioned (SSD, HDD, cloud type)
       ↓  auto-provisions
  PersistentVolume (PV)  →  actual storage resource (like a hard drive)
       ↑  claimed by
  PersistentVolumeClaim (PVC)  →  your app's request for storage ("I need 10GB SSD")
       ↑  mounted in
  Pod / Volume mount  →  /var/lib/data in your container

  Think of it like:
  StorageClass = Type of apartment building
  PV = Individual apartment unit (real physical space)
  PVC = Lease agreement (claiming that apartment)
  Pod = Tenant moving in

LESSON 11 — NETWORKING
INGRESS &
LOAD BALANCING
Ingress is a K8s-native HTTP router. It routes external traffic to different Services based on hostname or path — like Nginx but managed by Kubernetes.
INGRESS ROUTING
  Internet
     │
     ▼
  Ingress Controller (Nginx/Traefik/HAProxy running in K8s)
     │
     ├── yourdomain.com/api/*       →  Service: demo-app-service:80
     ├── yourdomain.com/reports/*   →  Service: reports-service:80
     ├── admin.yourdomain.com       →  Service: admin-service:80
     └── yourdomain.com/static/*    →  Service: cdn-service:80

  Each Service then load-balances to its Pods automatically.

& AUTO-SCALING
Tell Kubernetes how much CPU/memory your app needs. The HPA (Horizontal Pod Autoscaler) automatically scales replicas up/down based on CPU, memory, or custom metrics.

ℹ️ Requests vs Limits: requests = guaranteed resources (used for scheduling). limits = hard cap. If CPU exceeds limit, it's throttled. If memory exceeds limit, the container is OOM-killed and restarted.

DAEMONSETS
Beyond Deployments — specialized controllers for stateful apps like databases and node-level daemons.
ControllerUse caseKey difference from Deployment
DeploymentStateless apps (Spring Boot, Nginx)Pods are interchangeable, random names
StatefulSetStateful apps (databases, Kafka, Redis)Stable pod names (db-0, db-1), stable storage, ordered start/stop
