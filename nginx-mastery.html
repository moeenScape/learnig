LESSON 01 — INTRODUCTION
WHAT IS NGINX?
Nginx (pronounced "engine-x") was created in 2004 by Igor Sysoev to solve the C10K problem — handling 10,000 concurrent connections. Today it powers over 34% of all websites, including Netflix, Dropbox, GitHub, and WordPress.com.
🌐
Web Server
Serves static files (HTML, CSS, JS, images) directly to browsers — blazing fast.
🔀
Reverse Proxy
Sits in front of your app server. Forwards client requests, returns responses.
⚖️
Load Balancer
Distributes traffic across multiple app instances for high availability.
💾
HTTP Cache
Caches backend responses. Serves cached content without hitting your app.
🔒
SSL Terminator
Handles HTTPS/TLS so your app only needs to deal with plain HTTP internally.
🚪
API Gateway
Routes /api/* → backend, /static/* → CDN, /admin/* → separate service.

How Nginx Handles Requests Internally

NGINX EVENT-DRIVEN ARCHITECTURE
  Apache (thread-per-request):
  Request 1 → Thread 1 (blocked waiting for DB)
  Request 2 → Thread 2 (blocked waiting for DB)
  Request 3 → Thread 3 (blocked waiting for DB)
  1000 requests → 1000 threads → 💥 memory exhausted

  Nginx (event-driven, non-blocking):
  Master Process
  └── Worker 1 (handles ALL connections with event loop)
      ├── Connection 1: sent request to backend... (waiting, non-blocking)
      ├── Connection 2: received response, sending to client
      ├── Connection 3: reading request headers
      └── Connection 4: serving cached file
  Worker 2, Worker 3, Worker 4... (one per CPU core)

  Result: 10,000 connections → same memory as 10 connections ✅
ℹ️ Nginx uses the OS's epoll (Linux) or kqueue (macOS/BSD) event notification system — the same tech that powers Node.js's event loop. This is why both are so fast at I/O.

LESSON 03 — CONFIGURATION
CONFIG FILE
STRUCTURE
Everything in Nginx is configured via text files. Understanding the context hierarchy is the key to mastering Nginx. Settings in outer contexts apply to inner contexts unless overridden.
FILE SYSTEM LAYOUT
  /etc/nginx/
  ├── nginx.conf              ← Main config. Includes everything else.
  ├── sites-available/        ← ALL site configs (active or not)
  │   ├── default             ← Default Nginx welcome page
  │   └── demo-app              ← Your Spring Boot app config
  ├── sites-enabled/          ← Only ACTIVE configs (symlinks to sites-available)
  │   └── demo-app ──────────────► ../sites-available/demo-app
  ├── conf.d/                 ← Additional config snippets (auto-included)
  ├── snippets/               ← Reusable config fragments
  │   ├── fastcgi-php.conf
  │   └── snakeoil.conf
  ├── modules-enabled/        ← Loaded Nginx modules
  └── mime.types              ← File extension → Content-Type mapping

  Log files:
  /var/log/nginx/access.log   ← All HTTP requests
  /var/log/nginx/error.log    ← Errors and warnings

LESSON 07 — BEGINNER
LOCATION
BLOCKS
Location blocks are the most powerful and confusing part of Nginx. They match incoming request URIs and apply different rules. Understanding matching priority is critical.
LOCATION MATCHING PRIORITY (highest to lowest)
  Priority 1 (highest): location = /exact       Exact string match. Stops immediately.
  Priority 2:           location ^~ /prefix    Prefix match. Stops regex from running.
  Priority 3:           location ~ \.php$      Case-SENSITIVE regex match.
  Priority 3:           location ~* \.PHP$     Case-INSENSITIVE regex match.
  Priority 4 (lowest):  location /prefix       Regular prefix match (longest wins).

  ┌────────────────────────────────────────────────────────┐
  │  Request: GET /api/users/123                           │
  │                                                        │
  │  location = /api/users/123  → MATCHES (exact, wins!) │
  │  location ^~ /api/          → would match             │
  │  location ~ /api/\w+        → would match             │
  │  location /api/             → would match             │
  └────────────────────────────────────────────────────────┘
/etc/nginx/sites-available/demo-app — all location types
location matching test
⚠️ A common mistake: putting location / before more specific locations. Order matters for regex — Nginx checks regex locations top-to-bottom. Exact and prefix matches use longest-match logic regardless of order.

LESSON 09 — INTERMEDIATE
LOAD
BALANCING
When one app instance isn't enough, Nginx distributes traffic across multiple backends. The upstream block defines the server pool.
🔄
round_robin (default)
Requests rotate evenly: 1→server1, 2→server2, 3→server3, 4→server1...
⚡
least_conn
Sends to server with fewest active connections. Best for variable-length requests.
📌
ip_hash
Same IP always goes to same server. Maintains session affinity without shared sessions.
⚖️
weight
Proportional traffic. weight=3 gets 3x more requests than weight=1.

LESSON 11 — INTERMEDIATE
RATE
LIMITING
Protect your API from abuse, brute-force attacks, and DDoS. Nginx's rate limiting uses the token bucket algorithm — fast but fair.
TOKEN BUCKET ALGORITHM
  limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s

  Every IP has a "bucket" that fills at 10 tokens/second:
  ┌─────────────────────────────────────────────────────┐
  │  IP: 203.45.12.67  [ ●●●●●●●●●●●●●●●●●●●● ]  20   │  ← burst full
  └─────────────────────────────────────────────────────┘
  Request arrives → consume 1 token → OK ✅
  Request arrives fast (burst=20) → tokens available → OK ✅
  Requests keep coming (>20 burst, >10r/s sustained) → ❌ 429

  nodelay: serve burst immediately (don't queue)
  without nodelay: queue burst requests (adds latency)
/etc/nginx/nginx.conf + sites-available/human
rate limit simulation
⚠️ limit_req_zone MUST be in the http {} context (in nginx.conf or a file included from it). It cannot go inside a server {} or location {} block.

LESSON 12 — INTERMEDIATE
PROXY
CACHING
Nginx can cache responses from your backend, serving them instantly without hitting your Spring Boot app or Oracle database. This is the fastest possible response — from memory or disk cache.
CACHE FLOW
  Request #1  →  Nginx MISS  →  Spring Boot  →  Oracle DB  →  Response (cached)
  Request #2  →  Nginx HIT   →  Serve from cache (0ms backend time!) ⚡
  Request #3  →  Nginx HIT   →  Serve from cache ⚡
  ...10 minutes later...
  Request #N  →  Nginx EXPIRED → Spring Boot  →  Oracle DB  →  Response (re-cached)

  Check cache status: curl -I http://yourdomain.com/api/reports
  X-Cache-Status: HIT   ← served from cache
  X-Cache-Status: MISS  ← went to backend
  X-Cache-Status: BYPASS ← cache was bypassed

LESSON 15 — ADVANCED
SECURITY
HARDENING
Default Nginx is functional but not hardened. Apply these security layers to protect against common web attacks, information leakage, and abuse.
1
Hide Nginx version
Add server_tokens off to http context. Prevents attackers from knowing your exact Nginx version and targeting known CVEs.
2
Limit HTTP methods
Block methods your API doesn't use. Most REST APIs only need GET, POST, PUT, DELETE, OPTIONS.
3
Block bad user agents & scanners
Block known vulnerability scanners and bots with map directives.
4
Limit request body size
client_max_body_size 10m prevents memory exhaustion from large uploads unless you explicitly allow them.
5
Add security headers
HSTS, X-Frame-Options, X-Content-Type-Options, CSP — these are free security wins.
security hardened config
